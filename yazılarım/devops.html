DevOps Nedir, Neden Önemli?

DevOps (Development + Operations), yazılım geliştirme ve sistem operasyonları arasında işbirliği ve entegrasyonu artırmayı hedefleyen bir yaklaşımdır. Bu metodoloji, daha hızlı ve güvenilir yazılım teslimatını sağlar.

DevOps kültürü, geleneksel silo yapılarını kırarak geliştiriciler ve sistem yöneticileri arasında ortak sorumluluk anlayışını benimser. Bu yaklaşım otomasyonu, sürekli entegrasyonu ve sürekli teslimatı temel alır.

💡 Pro İpucu: DevOps sadece araçlar değil, bir kültür değişimidir. İnsanların düşünce biçimini ve çalışma şeklini değiştirmeyi hedefler.
DevOps'un Ana Faydaları:
  • • Daha hızlı yazılım teslimatı
  • • Daha az hata ve daha hızlı düzeltme
  • • Otomatik test ve deployment
  • • Daha iyi takım işbirliği
  • • Müşteri memnuniyetinde artış
DevOps Döngüsü
Plan → Code → Build → Test → Release → Deploy → Operate → Monitor - Bu süreç sürekli olarak tekrar eder ve iyileştirilir.

Amazon, Netflix, Google gibi büyük teknoloji şirketleri DevOps pratiklerini benimser. Bu şirketler günde yüzlerce deployment yapabilir ve sistemlerinin çalışabilirliğini yüksek seviyelerde tutar.

Docker Temelleri

Docker, uygulamaları containerlar içinde paketleyerek tutarlı ve taşınabilir çalışma ortamları sağlayan bir platformdur. Sanal makinelere kıyasla daha hafif ve verimli bir sanallaştırma sunar. 

# Dockerfile örneği - Node.js uygulaması
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000

CMD ["npm", "start"]
Dockerfile Talimatları
FROM - Base image
RUN - Build sırasında komut çalıştır
COPY - Host'tan dosya kopyala
WORKDIR - Çalışma dizini belirle
EXPOSE - Port bilgisi
CMD - Container başlatıldığında çalışacak komut

Image oluşturma ve çalıştırma: 

# Image build etme
docker build -t my-app:v1.0 .

# Image çalıştırma
docker run -p 3000:3000 my-app:v1.0

# Background'da çalıştırma
docker run -d -p 3000:3000 --name my-container my-app:v1.0
İpucu: .dockerignore dosyası kullanarak gereksiz dosyaları image'a dahil etmekten kaçının. Bu build süresini hızlandırır.
Best Practice: Multi-stage build kullanarak production image'larınızı mümkün olduğunca küçük tutun.

Docker Compose ile Multi-Container Uygulamalar

Docker Compose, çok container'lı uygulamaları tanımlamak ve çalıştırmak için kullanılan bir araçtır. YAML formatında yazılmış dosyalarla servislerinizi yönetir. 

# docker-compose.yml dosyası
version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - db
    environment:
      - NODE_ENV=production

  db:
    image: postgres:15
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=password
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:

Compose komutları:

# Servisleri başlatma
docker-compose up

# Background'da başlatma
docker-compose up -d

# Servisleri durdurma
docker-compose down

# Log'ları görme
docker-compose logs

# Belirli servis için log
docker-compose logs web
Docker Compose Avantajları:
  • • Çok container'lı uygulamaları tek komutla yönetme
  • • Servisler arası network otomatik oluşturma
  • • Environment variables yönetimi
  • • Volume ve network tanımlama
  • • Scaling imkanları
Servis İletişimi
Compose'da servisler birbirleriyle servis ismiyle iletişim kurabilir. Örnek: db:5432

Kubernetes (K8s) Giriş

Kubernetes, container'ları büyük ölçekte yönetmek için Google tarafından geliştirilen açık kaynaklı bir orkestrasyon platformudur. Mikroservis mimarilerin production ortamında çalıştırılmasında kritik rol oynar. 

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:v1.0
        ports:
        - containerPort: 3000
Kubernetes Ana Bileşenleri:
  • Pod: En küçük deployment birimi
  • Service: Pod'lara erişim sağlayan abstraction
  • Deployment: Pod'ların desired state yönetimi
  • Namespace: Kaynak gruplandırma
  • ConfigMap/Secret: Konfigürasyon yönetimi
Self-Healing
Kubernetes, container'ların sağlık durumunu izler ve başarısız olanları otomatik olarak yeniden başlatır. 
# Temel kubectl komutları
kubectl apply -f deployment.yaml
kubectl get pods
kubectl get services
kubectl describe pod my-app-xxx
kubectl logs my-app-xxx
kubectl scale deployment my-app --replicas=5
İpucu: Yerel geliştirme için Minikube veya Docker Desktop'ın Kubernetes desteğini kullanabilirsiniz.

CI/CD Pipeline Oluşturma

CI/CD (Continuous Integration/Continuous Deployment), kod değişikliklerinin otomatik olarak test edilip production'a deploy edilmesi sürecidir. GitLab CI, GitHub Actions, Jenkins gibi araçlar kullanılır. 

# .github/workflows/deploy.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
    - run: npm ci
    - run: npm test

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    - name: Build Docker Image
      run: docker build -t myapp:${{ github.sha }} .
    - name: Deploy to Production
      run: echo "Deploying to production..."
CI/CD Pipeline Aşamaları:
  • Source: Kod repository'den çekme
  • Build: Kodu compile etme, bağımlılıkları yükleme
  • Test: Unit, integration, e2e testleri
  • Security Scan: Güvenlik zafiyet taraması
  • Deploy: Staging/Production ortamına yerleştirme
Blue-Green Deployment
İki identik production ortamı kullanarak risk-free deployment yapma stratejisi. Hata durumunda anında geri dönüş mümkündür.
İpucu: Automated testing olmadan CI/CD implementasyonu risklidir. Test coverage'ınızı %80 üzerine çıkarmaya odaklanın.

Monitoring & Logging

Production sistemlerinin sürekli izlenmesi kritiktir. Prometheus, Grafana, ELK Stack gibi araçlar ile sistem metrikleri ve log analizleri yapılır. 

# docker-compose.yml - Monitoring Stack
version: '3.8'

services:
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml

  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin

  elasticsearch:
    image: elasticsearch:8.5.0
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
İzlenmesi Gereken Metrikler:
  • • Golden Signals: Latency, Traffic, Errors, Saturation
  • • Infrastructure: CPU, Memory, Disk, Network
  • • Application: Response time, Error rate, Throughput
  • • Business: Conversion rate, User satisfaction
Alerting
Kritik metrikler için threshold'lar belirleyin. Slack, PagerDuty, email ile otomatik alarm sistemi kurun.
İpucu: "Observability" sadece monitoring değildir. Metrics, Logs ve Traces'in (MLT) üçlü kombinasyonu gereklidir.

Container Security

Container güvenliği, DevOps süreçlerinin kritik bir parçasıdır. Image scanning, runtime protection ve access control konularında dikkatli olmak gerekir. 

# Security-focused Dockerfile
FROM node:18-alpine

# Non-root user oluşturma
RUN addgroup -g 1001 -S nodejs
RUN adduser -S nextjs -u 1001

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

COPY --chown=nextjs:nodejs . .

USER nextjs

EXPOSE 3000
CMD ["npm", "start"]
Güvenlik Uyarıları:
• Container'ları root user ile çalıştırmayın
• Base image'ları güncel tutun
• Gereksiz paketleri image'a dahil etmeyin
• Secret'ları environment variable olarak geçmeyin
Container Security Best Practices:
  • • Image Scanning: Trivy, Clair, Snyk kullanımı
  • • Runtime Security: Falco, Twistlock implementasyonu
  • • Network Policies: Kubernetes network segmentation
  • • RBAC: Role-based access control
  • • Secrets Management: Vault, AWS Secrets Manager
Secrets Yönetimi
API key'leri, password'ları asla kodda saklamayın. Kubernetes Secret, Docker Secret veya external vault sistemleri kullanın.

DevOps Best Practices

Başarılı DevOps implementasyonu için kanıtlanmış praktikler ve metodolojiler:

Infrastructure as Code (IaC):
  • • Terraform: Multi-cloud infrastructure provisioning
  • • Ansible: Configuration management
  • • CloudFormation: AWS native IaC
  • • Helm: Kubernetes package management
İpucu: "Everything as Code" prensibini benimseyin. Infrastructure, configuration, policy, documentation - her şey version control altında olmalı.
Immutable Infrastructure
Server'ları güncelleme yerine yenisini deploy edin. Bu consistency sağlar ve rollback kolaylaştırır. 
# Terraform örneği
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1d0"
  instance_type = "t3.micro"
  
  tags = {
    Name        = "WebServer"
    Environment = "Production"
  }
}
Yaygın Hatalar:
• Manual configuration changes yapma
• Monitoring ve logging'i sonradan düşünme
• Single point of failure bırakma
• Backup stratejisi olmadan production'a geçme

DevOps Araç Ekosistemi

DevOps süreçlerinde kullanılan temel araçlar ve kategorileri:

Kategori Bazında Araçlar:
  • Version Control: Git, GitLab, GitHub, Bitbucket
  • CI/CD: Jenkins, GitLab CI, GitHub Actions, CircleCI
  • Containerization: Docker, Podman, containerd
  • Orchestration: Kubernetes, Docker Swarm, Nomad
  • Infrastructure: Terraform, Ansible, Puppet, Chef
  • Monitoring: Prometheus, Grafana, Datadog, New Relic
  • Logging: ELK Stack, Fluentd, Splunk
  • Security: Vault, OWASP ZAP, Aqua Security
Cloud Providers
AWS: ECS, EKS, Lambda, CloudFormation
Azure: AKS, Container Instances, DevOps
GCP: GKE, Cloud Run, Cloud Build
İpucu: Araç seçiminde "vendor lock-in" riskini göz önünde bulundurun. Open source ve cloud-agnostic çözümleri tercih edin.

Container vs Virtual Machine
Containerlar host işletim sisteminin kernel'ini paylaşır, VM'ler ise her biri kendi işletim sistemine sahiptir. Bu nedenle containerlar daha hızlı başlar ve daha az kaynak kullanır.

Temel Docker Komutları:

# Image'ları listeleme
docker images

# Çalışan container'ları görme
docker ps

# Tüm container'ları görme
docker ps -a

# Container durdurma
docker stop container_name

# Container silme
docker rm container_name
💡 Pro İpucu: docker system prune komutuyla kullanılmayan image'lar, container'lar ve network'leri temizleyebilirsiniz.
Docker'ın Ana Bileşenleri:
  • Image: Container'ın şablonu
  • Container: Çalışan image örneği
  • Registry: Image deposu (Docker Hub)
  • Dockerfile: Image build talimatları
  • Volume: Kalıcı veri saklama
Dikkat: Container'lar varsayılan olarak stateless'tir. Önemli verileri volume veya bind mount kullanarak saklayın.
